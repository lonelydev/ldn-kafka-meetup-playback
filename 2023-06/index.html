Kafka enterprise strategy to evolve with demand

John Mille - Principal Cloud Engineer at Sainsbury Tech

Walk through Sainsbury tech's Kafka journey so far, from self managed on-premises to the cloud

Started off with little Kafka experience

Learning from others, tooling and documentation. Making many mistakes.

Key take-aways

  • Define rules to ensure data consistency
  • Define a topic naming convention that can be captured using a simple regex

DaaP - Data as a Product

  • Define a topic naming convention that can be captured using a simple regex - source/producer/owner at a glance
  • Categorise topics - identify public and private

Starters

  • Topics must have a version id
  • Public topics must have a schema
  • After the first message has been produced - cannot change the following
    • partition count (change can mess up ordering)
    • retention policy (could impact your mirroring solution - replicator/mirrormaker)

Tooling

  • Deployment automation of Kafka resources - whatever tech you use
  • Viewing your topics- many open source tools available - limited feature-set but good starters like AKHQ
  • Observability - monitoring and auditing

Enabling Kafka GitOps

  • Shared responsibility model
  • Solution that scales - making development teams self sufficient
  • Solutions agnostic to vendors and are Apache Kafka Native - makes it easy to move without too much rework - "the protocol is open source, so should the tooling"

Monitoring

  • Find the right metrics - governance as platform team, other metrics for business and for devs
  • Managed services should free your team from operations
  • As admins:
    • keep an eye on topics and partition count
    • ingress and egress - io in message count and bytes and number of connections
    • Storage requirements - usage trends

Monitoring for developers

  • Offset Lag - per consumer per topic per partition, your consumer app performance, scaling needs based on workload
  • Message counts and size - evaluate if Kafka is really a good choice

For C-Suite

    Number of meters, accounts, cost per meter, forecasting data, etc.

Tooling

  • UI: Conduktor Console - highly recommended - web based, actively maintained, licensed.
  • cfn-kafka-admin - CLI and Lambdas to CRUD kafka resources via Cloud Formation

Tooling

  • Monitoring tools
    • Kafka lag exporter - highly efficient tool to estimate latency
    • Kafka Connect Watcher - custom tool developed to monitor connect clusters and send alerts where appropriate

Don't reinvent the wheel: use Kafka Connect and connectors

  • Open source connectors are maintained by multiple vendors
  • There are vendor specific connectors available as a paid service too

I like reinventing wheels though ü§™

  • Be prepared to handle failures, restarts, logging, scaling, serialisation and various data formats
  • You get to contribute, find problems and fix bugs yourself, maintain it
  • Could have invested that effort building your actual business related features instead

Did it all help?

No production data corruption incidents!

Teams have the tools they need and are self sufficient when it comes to using Kafka!

Bad practices can be identified and rectified early as Ops team have visibility!

What next For John's team?

  • Common framework for every consumer/producer to go through similar access patterns - common suite of testing practices
  • Data Cataloging and data discovery made easier
  • Kafka best practice education

how do we get there?

  • Finding the right tool to make it easier to enforce best practices - Exploring Conduktor Gateway
  • Creating a Kafka admins/maintainers group - we have a kafka caretakers group here today
  • Share deployment methods and techniques among business units - Sainsbury's have several subdivisions

Summary

  • Adopting kafka resource lifecycle rules pays off
  • Monitor what you need - not everything (could get expensive very quickly)
  • Prefer vendor agnostic tooling
  • Create an internal Kafka practices Community
  • Share learnings outside your org

Some Q&A from the session

  • Your architecture determines your cost - design with the cost in mind. Don't use streaming if you really only do batch processing.
  • Datamasking, obfuscation and encryption on the fly
